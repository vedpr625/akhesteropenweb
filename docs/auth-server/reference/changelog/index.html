Changelog

Version 0.2.3 — Code quality and correctness​

Bug fixes​

SmartTokenCustomizer — launch token extraction API corrected

The launch token was being extracted using context.getAuthorization().getAttribute("additional_parameters") — an internal Spring Authorization Server implementation detail that is not part of the public API and may not be populated in all versions. Fixed to use the documented stable API:

OAuth2AuthorizationRequest authRequest = context.getAuthorization()
        .getAttribute(OAuth2AuthorizationRequest.class.getName());

This is the correct Spring Authorization Server 1.3.x pattern for accessing the original authorization request during token customization.

RegisteredAppRepository duplicate interface removed

JpaRegisteredClientRepository.java contained a package-private inner interface RegisteredAppRepository that duplicated the standalone RegisteredAppRepository.java. Spring Data JPA would find two candidates for the same repository type. The inner interface has been removed; only the standalone file remains.

Improvements​

DataInitializer — seeds registered apps on startup

DataInitializer now injects RegisteredAppRepository and seeds the ajfhir-smart-client app registration on first startup alongside the test clinicians. Previously the seeding was a commented-out TODO. Both seeding methods are idempotent.

SmartServerProperties — SMART client URL is now configurable

LaunchPortalController previously had the SMART client launch URL hardcoded as "http://localhost:8080/launch" — a private static final constant that made the server impossible to deploy without source changes. Added smart.server.smart-client-launch-url (env: SMART_CLIENT_LAUNCH_URL) to SmartServerProperties. Defaults to http://localhost:8081/launch for local development.

SmartDiscoveryProxyFilter — 60-second in-memory cache

The discovery proxy now caches successful responses for 60 seconds using an AtomicReference<CachedResponse>. The SMART discovery document almost never changes between deployments, so making an outbound HTTP call on every SMART launch was unnecessary. Cache is thread-safe with no locking overhead. Error responses (non-200) are not cached.

Version 0.1.0 — Initial build​

Built task-by-task alongside the ajfhir-smart-client client. Designed to be the auth server the client talks to instead of Epic.

Core server scaffold​

Files: pom.xml, SmartFhirServerApplication, SmartServerProperties, application.yml, application-dev.yml

  • Spring Boot 3.3.5, Spring Authorization Server 1.3.x, Java 21
  • SmartServerProperties — @ConfigurationProperties(prefix="smart.server") with @NotBlank, @Min(60) validation
  • H2 dev profile — application-dev.yml with in-memory database, H2 console enabled
  • @EnableScheduling on main app — activates @Scheduled on LaunchContextService.purgeExpiredTokens()

RSA key pair and JWKS​

Files: RsaKeyConfig, JwksController

  • RSA-2048 key pair generated at startup via Nimbus RSAKeyGenerator
  • JWKSource<SecurityContext> bean for Spring Authorization Server JWT signing
  • GET /oauth2/jwks — returns public key only via rsaKey.toPublicJWK() — never exposes d, p, q
  • Key ID is a random UUID — matches across discover and verify

Spring Authorization Server configuration​

Files: AuthorizationServerConfig

Critical fixes applied:

  • Added OAuth2AuthorizationService bean (InMemoryOAuth2AuthorizationService) — without this, every token exchange returns invalid_grant
  • Added OAuth2AuthorizationConsentService bean — without this, the authorize endpoint throws NullPointerException
  • Registered SmartTokenResponseConverter as accessTokenResponseHandler on the token endpoint — enables top-level SMART extras in response body
  • H2 console permitted in security config and CSRF disabled for /h2-console/**

Clinician user store and app registry​

Files: Clinician, ClinicianRepository, ClinicianUserDetails, CliniciansUserDetailsService, RegisteredApp, JpaRegisteredClientRepository, DataInitializer

  • Clinician JPA entity — username, BCrypt password hash, display name, FHIR Practitioner ID
  • JpaRegisteredClientRepository — replaces InMemoryRegisteredClientRepository, loads apps from PostgreSQL at runtime
  • RegisteredApp — allowedScopeList() parses comma-separated scope string
  • DataInitializer — seeds dr.smith, dr.jones, and ajfhir-smart-clientjfhir-smart-client` on first startup

Launch token system​

Files: LaunchContext, LaunchContextRepository, LaunchContextService, LaunchPortalController, LaunchTokenException

  • LaunchContext JPA entity — single-use, 5-minute expiry, used flag
  • LaunchContextService.resolveLaunchToken() — marks used before returning (prevents race condition replay)
  • @Scheduled purge every 10 minutes
  • LaunchPortalController — fetches patients from HAPI FHIR, redirects to ?iss=&launch=

SMART token customization​

Files: SmartTokenCustomizer, SmartTokenResponseConverter

Critical fix applied:

  • SmartTokenCustomizer — fixed launch param extraction from OAuth2AuthorizationRequest.getAdditionalParameters().get("launch") — previous approach using "additional_parameters" attribute key was incorrect for Spring Authorization Server 1.3.x
  • SmartTokenResponseConverter — promotes patient, encounter, need_patient_banner from JWT claims to top-level token response body fields via SignedJWT.parse() + JWTClaimsSet

SMART discovery​

Files: SmartDiscoveryController, SmartDiscoveryProxyFilter

  • SmartDiscoveryController — returns exact fields SMART client's SmartConfiguration record maps: authorization_endpoint, token_endpoint, capabilities (including launch-ehr), code_challenge_methods_supported: [S256], jwks_uri
  • SmartDiscoveryProxyFilter — OncePerRequestFilter for HAPI server — forwards /.well-known/smart-configuration to auth server

Scope enforcement​

Files: SmartScopeAuthorizationInterceptor

  • HAPI @Interceptor registered on HAPI FHIR server (not this auth server)
  • Validates RS256 JWT signature using server's public RSA key (Nimbus DefaultJWTProcessor)
  • Parses SMART v2 scope grammar: patient/{ResourceType}.{ops} and patient/*.{ops}
  • Maps HTTP methods to scope operation letters (GET→r, POST→c, PUT→u, DELETE→d)

UI​

Files: login.html, portal.html

  • Thymeleaf login page with clinical blue styling
  • Patient picker table — name, DOB, gender, FHIR ID, Launch button per row
  • Patient search form — GET /portal?search=Smith

Tests​

Files: LaunchContextServiceTest, ServerComponentTests, AuthorizationServiceBeansTest

  • LaunchContextServiceTest — 9 tests covering token creation, resolution, single-use enforcement
  • ServerComponentTests — RSA key tests, JWKS public-only, scope permission logic
  • AuthorizationServiceBeansTest — OAuth2AuthorizationService beans, SmartDiscoveryProxyFilter path matching
